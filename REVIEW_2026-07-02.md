## Executive summary

- **Overall health: good.** This is a well-tended codebase. The auth gate, tenancy model,
  CSRF, secret-boot-guard, compression, server-side Discover cache, and clientside row-highlight
  work are all real and correctly done. Most of what a cold review would flag has already been
  found and fixed by the two existing plans. The team's discipline (dated fix log, "rejected with
  reason" entries) is a strength worth preserving.
- **Top 3 risks:**
  1. **Deploy config contradicts the code's own concurrency assumptions.** `cloudbuild.yaml`
     runs `--max-instances=4` with no `CACHE_REDIS_URL`, but the SimpleCache, the chat
     rate-limiter, and the `/internal/cache/refresh` endpoint are all explicitly documented as
     "fine only at one instance." Not a data leak — but cache refresh reaches 1 of 4 instances,
     the rate limit is effectively 4×, and each instance re-runs the full BQ preload. (F1)
  2. **`assets/detail_debug.js` is live.** A "TEMPORARY DEBUG — delete this file" script that
     monkeypatches `window.fetch` is sitting in `assets/`, which Dash auto-injects into *every*
     dashboard page, and `.dockerignore` doesn't exclude it — so it ships and runs in prod. (F2)
  3. **Multi-tenant cache isolation is structurally unkeyed** — safe *today* only because every
     dashboard is bespoke-single-client. The day one dashboard serves two clients, the
     process-global `data_manager` cache serves client A's rows to client B. Already documented
     (§5.3); restated here with its trigger. (F8)
- **Top 3 quick wins:**
  1. Delete `assets/detail_debug.js` and commit the already-staged deletions
     (`charts.py`, `_smoke_callback.py`, `claude-chats-debug.log`, `ext_crossfilter.*`,
     `charts_shared.py`). (F2, F3)
  2. Close `.dockerignore` gaps so `.claude/worktrees/` (1.2 MB duplicate source tree),
     `notepad.md`, `test-results/`, and `memory/` stop shipping in the prod image. (F4)
  3. Untrack `notepad.md` and `test-results/.last-run.json` (the latter contradicts the
     `test-results/` line already in `.gitignore`). (F5)

## Speed

No new hot-path finding survived verification — the expensive layers were already addressed:

- **Compression:** `Compress(server)` is engaged (`app.py:374`); figure/table JSON ships gzipped.
- **Data round-trips:** Discover keeps records in a server-side cache (`charts_discover.py:106`)
  instead of round-tripping a `dcc.Store`; row highlight is clientside; detail-figure rebuilds
  remain server callbacks but read from warmed caches, so the cost is JSON serialization +
  network, not BigQuery. That's the right split for this data shape.
- **Boot / first paint:** pages for all dashboards build eagerly at import (single-Vizro-app
  constraint, §5.21) but `warm_caches()` runs the BQ preload on a **daemon thread**
  (`amazon_2026/__init__.py:318`), so boot doesn't block on network. On `min-instances=0` the
  cold-start tax is real but bounded (8-worker preload pool).
- **Only measured regression is F1's cold-start multiplication** (each of up to 4 instances
  re-preloads). Fixing F1 removes it. I did **not** invent latency numbers; the existing plan's
  measurement (server-side render dominates) still holds and no clientside conversion is missing
  on the hot path.

## Verified fine — no action (so these aren't re-flagged)

- **Auth gate** (`app.py:263`): guards everything under the mount; static assets exempted
  deliberately; `/ext/chat/ask` self-gates with `can_access`. Sound.
- **Impersonation** (`auth/middleware.py:94`): honored only when the *real* user is admin;
  refuses admin targets and disabled accounts. Correct.
- **CSRF** (`security.py`): synchroniser-token on every state-changing form; JSON/token
  endpoints correctly exempt. `/sessionLogin` is safe (requires unguessable Firebase ID token).
- **Secret boot guard** (`app.py:365`): refuses prod boot on the default session secret.
- **BQ cost cap**: `BQ_MAX_BYTES_BILLED=20000000000` set in prod deploy; `run_query` honors it.
- **`safe_query`** (`data_sources/bq.py:61`): fixture fallback only in dev, re-raises in prod —
  never serves fabricated numbers as real client data.
- **Firebase client API key in `cloudbuild.yaml`**: that's a *public* client key by design, not
  a secret. Not a finding.
- **Dependency pins**: all resolve and match the installed `.venv`. Not stale.
- **Usage/audit recording**: fail-soft and off-thread; can't break the triggering request.
